# Release Notes Automation

`/csm/core/updates` 에 입력 가능한 카드 초안을 자동 생성하는 규칙.

## 트리거

이번 세션에서 `src/` 하위 파일에 **사용자 인지 가능한 변경**이 발생한 경우, 작업 응답의 마지막에 release-note 카드 초안을 마크다운으로 출력한다.

**대상:** 버그 수정, 기능 추가, UX 개선, 보안 패치, 운영 변경(설정/배포), API/스키마 변경.

**건너뛰는 경우:**
- typo / 주석만 수정
- 코드 포맷팅 / 정렬 변경
- 내부 리팩터링 (외부 동작 변화 없음)
- 테스트만 추가 (기능 변경 없음)
- 사용자가 명시적으로 "초안 필요 없음" 또는 "skip release note" 라고 한 경우

## 버전 번호 결정 (SemVer)

직전 게시 버전을 기준으로 다음 규칙을 적용한다.

| 변경 성격 | 증가 자리 | 예시 (v3.4.0 기준) |
|----------|----------|---------------------|
| 버그 수정만 | patch | v3.4.1 |
| 신규 기능 / UX 개선 | minor | v3.5.0 |
| Breaking change / API 호환성 깨짐 | major | v4.0.0 |

**한 세션에 여러 성격이 섞이면:** 가장 높은 자리를 따른다. 예: fix + feature → minor.

**직전 버전 파악 순서:**
1. 사용자가 이번 세션에서 명시한 버전을 우선 사용
2. 명시 없으면 다음 중 하나로 추정:
   - `git log --oneline -20` 의 commit 메시지에서 버전 태그 검색
   - `core_update` 테이블 SELECT 가능하면 최신 row 참조
   - 위 모두 불가하면 "**직전 버전 확인 필요**" 표시 후 사용자에게 질문
3. 추정 버전 사용 시 카드 상단에 `> 직전 버전 v3.4.0 가정` 같은 주석 한 줄 명시

## 필드 기본값

| 필드 | 기본값 | 비고 |
|------|--------|------|
| **분류 (category)** | 변경 성격 기반: `release` (기능), `maintenance` (운영/점검), `fix` (버그) | 혼합이면 가장 비중 큰 것 |
| **강조도 (severity)** | `normal`. 단, 보안/데이터 손실 위험·breaking → `critical`. 신규 기능·중요 개선 → `important` | |
| **상태 (status)** | `PUBLISHED` 추천 (DRAFT 가능) | 사용자가 명시하지 않으면 PUBLISHED |
| **팝업 (popup_yn)** | `N`. 단, `critical` 또는 사용자가 "팝업으로 띄워줘" 라고 명시 시 `Y` | |
| **공개일 (published_at)** | 작업 완료 시각 (KST, `YYYY-MM-DD HH:mm` 형식) | |

## 출력 형식

응답 마지막에 다음 형식으로 출력한다. 사용자가 그대로 `/csm/core/updates` 폼에 복사-붙여넣기할 수 있게 한다.

````markdown
---

### 📋 Release Note 초안 (vX.Y.Z)

> 직전 버전 vA.B.C 가정 — 다르면 알려주세요.

| 필드 | 값 |
|------|-----|
| **버전** | `vX.Y.Z` |
| **분류** | 신규 기능 (release) / 점검 (maintenance) / 버그 수정 (fix) |
| **강조도** | 일반 (normal) / 중요 (important) / 긴급 (critical) |
| **상태** | 게시 (PUBLISHED) |
| **팝업** | ❌ 또는 ✅ |
| **공개일** | `YYYY-MM-DD HH:mm` |

**제목**
```
한 문장 (60자 내)
```

**한 줄 요약**
```
고객 입장의 한 문장 요약 (200자 내)
```

**상세 내용**
```
주요 변경 / 버그 수정 / 운영 안내 등 섹션 구분
```
````

## 작성 톤

- **고객 시점**: "코드를 고쳤다"가 아니라 "무엇이 달라졌다"
  - ❌ "UserApiController.toLong 메서드의 JSON String 거부 문제 수정"
  - ✅ "역할 수정 후 저장이 반영되지 않던 문제 해결"
- **기술 용어는 필요할 때만**: 운영자가 알아야 하는 환경변수·재시작 안내는 "운영 안내" 섹션으로 분리
- **요약은 한 문장**: 상세 내용은 줄바꿈 OK, 요약은 한 줄
- **버그 수정에는 원인 → 해결 한 줄씩**: 운영자가 재현·검증할 수 있을 만큼만

## 동일 세션 내 누적

같은 세션에서 여러 작업이 진행된 경우 **하나의 카드로 합치는 것을 우선한다.** 단, 다음 경우엔 분리 제안:

- 변경 성격이 명확히 다른 경우 (예: 보안 패치 + UX 개편)
- 사용자가 별도 안내가 필요하다고 명시한 경우

## 사용자 보조 안내
